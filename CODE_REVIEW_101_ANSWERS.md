# Code Review 101 — 回答ガイド

先に自分のレビューを書いてから確認してください。表現が一致しなくても、問題の事実・発生条件・影響・実行可能な提案が含まれていれば正解です。

判定の目安：`Request changes` は障害・データ破壊・漏えいなどマージ前に直す問題、`Comment` は仕様確認や保守性の改善、`Approve` は変更不要です。

## 第1章：名前と意図

### 問001
**判定：Comment**
**回答例**：「`calc(a, b)` では計算目的と引数の意味が分かりません。単価×数量を計算する仕様なら `calculate_total_price(unit_price, quantity)` に変更してください。」
**確認・テスト**：何を計算し、入力と戻り値の単位が何かを仕様で確認する。

### 問002
**判定：Comment**
**回答例**：「`x` では先頭ユーザーであることが伝わらないため、`first_user` が適切です。」
**確認・テスト**：先頭要素を選ぶこと自体が仕様か確認する。

### 問003
**判定：Comment**
**回答例**：「`do_it` では処理内容を予測できません。例えば検証なら `validate_order(data)` のように、動詞と対象を名前に含めてください。」
**確認・テスト**：関数の副作用と戻り値を確認してから命名する。

### 問004
**判定：Comment**
**回答例**：「`flag` では何が真なのか不明です。メール送信済みを表すなら `is_email_sent = True` としてください。」
**確認・テスト**：真偽が表す状態を仕様で確認する。

### 問005
**判定：Comment**
**回答例**：「取得とメール送信という二つの責務が一つの関数にあり、取得だけを再利用・テストできません。`get_user` と `send_mail` に分け、呼び出し側で組み合わせてください。」
**確認・テスト**：ユーザー未検出時にメールを送らないテストを追加する。

### 問006
**判定：Comment**
**回答例**：「`86400` の意味と単位が不明です。1日の秒数なら `SECONDS_PER_DAY = 24 * 60 * 60` と定数化してください。」
**確認・テスト**：秒、ミリ秒など期待する単位を確認する。

### 問007
**判定：Comment**
**回答例**：「`check` では判定内容が不明です。`can_legally_drink(age)` や `is_drinking_age(age)` のように戻り値が真偽であることも示してください。」
**確認・テスト**：年齢基準は地域や日付により変わるため、20を固定してよいか確認する。

### 問008
**判定：Comment（仕様確認）**
**回答例**：「`process` が何を行い、`result` にどの型・状態が返る仕様でしょうか。失敗時が例外か `None` かも確認したいです。」
**確認・テスト**：入力形式、成功時の戻り値、失敗時の契約を確認する。

### 問009
**判定：Comment**
**回答例**：「`Manager` だけでは管理対象が分かりません。接続管理なら `ConnectionManager` のように責務を具体化してください。」
**確認・テスト**：複数の無関係な責務を持っていないかも確認する。

### 問010
**判定：Comment**
**回答例**：「`f` と `l` は意図を伝えず、`l` は数字の1とも見分けにくいです。`def count_items(items): return len(items)` のようにしてください。」
**確認・テスト**：単なる `len()` のラッパーが本当に必要かも確認する。

## 第2章：境界値と条件分岐

### 問011
**判定：Request changes**
**回答例**：「20歳以上を許可する仕様に対して `age > 20` では20歳を拒否します。`age >= 20` に変更してください。」
**テスト**：19、20、21歳。

### 問012
**判定：Comment**
**回答例**：「下限・範囲内・上限を確認するため、0、50、100をテストします。範囲外も含めるなら -1 と101も必要です。」
**テスト**：最低限0、50、100。入力検証では-1、101も追加する。

### 問013
**判定：Request changes（空入力があり得る場合）**
**回答例**：「`items=[]` のとき `items[0]` が `IndexError` になります。空の場合の戻り値または例外を仕様化し、アクセス前に処理してください。」
**修正例**：`first = items[0] if items else None`。空を異常とするなら明示的に例外を送出する。

### 問014
**判定：Request changes**
**回答例**：「`count == 0` で `ZeroDivisionError` が発生します。データ0件時の平均を `None`、0、例外のどれにするか決めて処理してください。」
**テスト**：countが0、1、複数。負数を許すかも確認する。

### 問015
**判定：Comment（仕様次第でRequest changes）**
**回答例**：「`'   '` は真なので空白だけの名前も保存されます。空白名を禁止するなら `if name.strip():` とし、保存する値も正規化してください。」
**テスト**：`''`、空白のみ、前後空白付き、通常名。

### 問016
**判定：Request changes**
**回答例**：「`range(len(items)+1)` は最後に `i == len(items)` となり、存在しない添字へのアクセスで失敗します。`range(len(items))` または直接 `for item in items` を使ってください。」
**テスト**：空、1件、複数件。

### 問017
**判定：Comment（仕様確認）**
**回答例**：「現在の条件では0度で `freeze()` は呼ばれません。0度を凍結扱いにするかは対象物や仕様で異なるため、`< 0` と `<= 0` のどちらかを明記してください。」
**テスト**：-1、0、1度。

### 問018
**判定：Comment（仕様次第でRequest changes）**
**回答例**：「現在は1000円ちょうどの割引が0です。『1000円以上』が対象なら `amount >= 1000` に変更してください。」
**テスト**：999、1000、1001円。

### 問019
**判定：Request changes（空入力があり得る場合）**
**回答例**：「`users=[]` では `users[-1]` が `IndexError` になります。ユーザーが0件の場合の扱いを決め、事前に空を処理してください。」
**テスト**：0件、1件、複数件。

### 問020
**判定：Request changes**
**回答例**：「`month <= 12` だけでは0や負数も通ります。月の範囲なら `1 <= month <= 12` としてください。」
**テスト**：0、1、12、13。

## 第3章：None・型・戻り値

### 問021
**判定：Request changes**
**回答例**：「未検出時に `find_user` が `None` を返すなら、`user['name']` で `TypeError` になります。未検出を分岐するか、取得関数が専用例外を返す契約にしてください。」
**テスト**：存在するIDと存在しないID。

### 問022
**判定：Comment（契約確認）**
**回答例**：「Pythonでは `add('1', '2')` が `'12'` となり、数値加算のつもりでもエラーになりません。数値専用なら型注釈と入力検証を追加してください。」
**テスト**：整数、浮動小数、文字列、混在型。

### 問023
**判定：Comment**
**回答例**：「`bool('false')` は文字列が空でないため `True` です。文字列の内容を真偽値として解釈するには、小文字化して許可値と明示的に比較してください。」
**修正例**：`value.strip().lower() == 'true'`。`true/false` 以外はエラーにする。

### 問024
**判定：Request changes**
**回答例**：「戻り値注釈は `list[str]` ですが実際には `None` を返しており、呼び出し側の反復処理が失敗します。常にリストを返すなら `[]`、未取得を表すなら `list[str] | None` にしてください。」
**テスト**：ユーザーあり・なし。

### 問025
**判定：Request changes（外部入力の場合）**
**回答例**：「`user_input='12.5'` や `'abc'` では `ValueError` になります。利用者入力なら変換失敗を捕捉して入力エラーとして返してください。」
**テスト**：`'42'`、`'abc'`、空文字、空白、負数。

### 問026
**判定：Request changes**
**回答例**：「`count` が欠落すると `get` は `None` を返し、`None + 1` で `TypeError` になります。欠落が0を意味するなら `data.get('count', 0) + 1`、必須なら明示的に検証してください。」
**テスト**：キーあり、キーなし、値がNone、値が文字列。

### 問027
**判定：Comment**
**回答例**：「計算関数が数値でなく文字列を返すと、後続の計算が連結や型エラーになります。表示用変換は呼び出し側に任せ、`a / b` の数値を返してください。」
**テスト**：整数除算、浮動小数、0除算。

### 問028
**判定：Comment**
**回答例**：「`None` は値の等価比較ではなく同一性で判定するため、`if value is None:` が推奨です。」
**確認**：`==` を独自実装したオブジェクトでも意図しない比較処理を呼ばない。

### 問029
**判定：Request changes**
**回答例**：「型注釈は実行時に `'42'` を整数へ変換しません。現在の値は文字列のままなので、`user_id = int(raw_user_id)` のように境界で変換してください。」
**テスト**：変換可能な文字列と変換不能な文字列。

### 問030
**判定：Request changes**
**回答例**：「成功時は辞書、失敗時はboolで、呼び出し側が毎回型分岐を必要とし見落とすと失敗します。結果型を統一するか、失敗は専用例外で表してください。」
**修正例**：常に `{'ok': bool, 'error': str | None}` を返す。より厳密にはdataclass等を使う。

## 第4章：例外処理

### 問031
**判定：Request changes**
**回答例**：「裸の `except` がシステム終了や割込みまで握りつぶし、失敗した処理を成功したように継続します。想定する例外だけを捕捉し、記録・通知・再送出を行ってください。」
**テスト**：想定例外が処理され、想定外例外は伝播すること。

### 問032
**判定：Request changes**
**回答例**：「すべての例外を `None` に変換すると、未検出・通信障害・実装バグを区別できず、スタックトレースも失われます。捕捉対象を限定し、原因を保持してください。」
**テスト**：各失敗原因が別の結果または例外になること。

### 問033
**判定：Request changes**
**回答例**：「ファイル不在は初期値作成で回復できる場合がありますが、JSON破損はデータ損失や不完全書込みを示します。同じ空設定として続行すると破損を隠すため、`FileNotFoundError` と `JSONDecodeError` を分けてください。」
**テスト**：不在、空、破損、正常JSON。

### 問034
**判定：Comment**
**回答例**：「汎用 `Exception` では呼び出し側が失敗原因を安全に判別できません。入力不正なら `ValueError`、業務固有なら専用例外を使ってください。」
**確認**：例外文字列の解析に依存しない設計にする。

### 問035
**判定：Request changes（重大）**
**回答例**：「例外メッセージはログや監視へ送られるため、パスワードを含めると認証情報が漏えいします。秘密値を除去またはマスクしてください。」
**テスト**：エラー時のログにパスワードやトークンが出ないこと。

### 問036
**判定：Request changes**
**回答例**：「`file` の代入前に失敗すると `finally` の `file.close()` が `UnboundLocalError` となり元の例外を隠します。`with open(...) as file:` を使用してください。」
**テスト**：open失敗時と処理途中失敗時。

### 問037
**判定：Request changes**
**回答例**：「恒久障害でも終了せず、CPU・回線・外部APIを消費し続けます。最大回数、指数バックオフ、対象例外、総時間上限を設けてください。」
**テスト**：最大回数で停止し、回復時には成功すること。

### 問038
**判定：Comment（通常は再試行しない）**
**回答例**：「401は認証情報が無効であることが多く、同じ資格情報で再試行しても成功せず負荷を増やします。トークン更新が可能な場合だけ更新後に1回再試行してください。」
**テスト**：更新成功、更新失敗、更新不能。

### 問039
**判定：Comment**
**回答例**：「503は一時的障害の可能性があります。冪等な操作に限定し、`Retry-After` を尊重して、指数バックオフと回数上限付きで再試行してください。」
**テスト**：503後の回復、上限到達、非冪等操作。

### 問040
**判定：Comment**
**回答例**：「同じ例外を各層で記録すると同一障害が重複計上され、アラート件数や調査を混乱させます。処理する層は文脈を加え、最上位など一か所でスタックトレースを記録してください。」
**確認**：ログ件数と相関IDで追跡できること。

## 第5章：関数と状態

### 問041
**判定：Comment**
**回答例**：「`global total` はテスト間で状態が残り、実行順序や並行実行で結果が変わります。現在値を引数で受け、新しい合計を返してください。」
**テスト**：各テストが単独・順不同で同じ結果になること。

### 問042
**判定：Comment**
**回答例**：「現在時刻を直接読むと境界時刻でテストが不安定になります。`now` または時計関数を引数で注入してください。」
**テスト**：期限直前、期限ちょうど、期限直後を固定時刻で確認する。

### 問043
**判定：Comment**
**回答例**：「環境変数を内部で読むと入力依存が見えず、テストごとの書換えも必要です。起動時に読み、検証済み設定を引数へ渡してください。」
**テスト**：設定あり、欠落、不正値。

### 問044
**判定：Request changes**
**回答例**：「可変デフォルトのリストは呼び出し間で共有され、2回目の結果に1回目の要素が残ります。既定値を `None` にし、関数内でリストを生成してください。」
**修正例**：`def add(item, items=None): items = [] if items is None else items`。

### 問045
**判定：Comment（仕様次第でRequest changes）**
**回答例**：「関数内の `items.sort()` は呼び出し元のリストも並べ替え、後続処理の順序を壊す可能性があります。副作用が契約でないなら `sorted(items)` で新しいリストを返してください。」
**テスト**：呼び出し後も元リストが変化しないこと。

### 問046
**判定：Comment**
**回答例**：「同じ規則が3か所にあると、仕様変更時に一部だけ直して結果が不一致になります。意味が同じ処理なら一つの関数へ抽出してください。」
**確認**：見た目が同じだけで仕様が異なる処理を無理に共通化しない。

### 問047
**判定：Comment**
**回答例**：「検証・保存・通知をそれぞれ関数へ分けると、保存失敗時に通知しないなどの経路を個別にテストできます。まず `validate`、`save`、`notify` が分割候補です。」
**テスト**：検証失敗、保存失敗、通知失敗。

### 問048
**判定：Request changes（更新が意図されない場合）**
**回答例**：「`get_user` という名前から読み取り専用と期待しますが、DBを更新するため、呼ぶだけで状態が変わります。取得と更新を分離するか、副作用を示す名前にしてください。」
**テスト**：取得操作だけではDBが変化しないこと。

### 問049
**判定：Request changes**
**回答例**：「失敗経路だけ暗黙に `None` を返すと、呼び出し側が成功値として扱う可能性があります。全経路で戻り値を明示し、失敗は結果型または例外で統一してください。」
**テスト**：成功・各失敗経路の戻り値契約。

### 問050
**判定：Comment**
**回答例**：「URLやタイムアウトを関数内に固定すると、本番・テスト・開発環境を切り替えられず、テストが実サービスへ接続します。設定または引数として渡してください。」
**テスト**：テスト用設定が利用されること。

## 第6章：コレクションとループ

### 問051
**判定：Request changes**
**回答例**：「走査中に要素を削除すると後続要素が左へ詰まり、ループの次の添字でその要素を飛ばします。条件に合わない要素から新しいリストを作ってください。」
**修正例**：`items = [item for item in items if not should_remove(item)]`。

### 問052
**判定：Request changes（順序が仕様の場合）**
**回答例**：「`set` は元の順序を保証しないため、重複除去後の表示順が変わります。順序を維持するなら `list(dict.fromkeys(items))` を使ってください。」
**テスト**：重複を含む入力で順序が維持されること。

### 問053
**判定：Comment（契約次第）**
**回答例**：「`name` が欠落すると `KeyError` です。必須なら事前検証または明示的なエラー、任意なら `data.get('name')` と欠落時の処理を使用してください。」
**テスト**：キーあり、なし、None、空文字。

### 問054
**判定：Comment**
**回答例**：「1万件を全件リスト化するとメモリ使用量と初回応答時間が増えます。全件が不要ならページング、順次処理ならジェネレーターを使ってください。」
**確認**：1万件が実際に問題か計測し、上限も確認する。

### 問055
**判定：Comment（規模によりRequest changes）**
**回答例**：「全ユーザー×全注文の比較は O(U×O) です。注文を `user_id` で辞書化し、一回の走査で照合すれば概ね O(U+O) にできます。」
**テスト**：重複ID、注文なし、大量データ。変更前後の性能も計測する。

### 問056
**判定：Request changes**
**回答例**：「入力が空だとループ内で `result` が代入されず、返却時に `UnboundLocalError` になります。空入力の戻り値を決め、事前処理または初期値を用意してください。」
**テスト**：空、1件、複数件。

### 問057
**判定：Comment**
**回答例**：「3段内包表記は短さより、処理順序・条件・変数の意味を誤読しないかを重視します。説明が必要なら通常のループまたは名前付き関数へ分けてください。」
**確認**：分割後も同じ結果になり、性能劣化が問題ないこと。

### 問058
**判定：Request changes**
**回答例**：「辞書同士には一般的な大小関係がなく `TypeError` になります。名前順なら `sorted(users, key=lambda user: user['name'])` のようにキーを指定してください。」
**テスト**：キー欠落、同値、昇順・降順。

### 問059
**判定：Comment（データ損失ならRequest changes）**
**回答例**：「同じIDを辞書へ代入すると先の値が無言で上書きされます。重複が異常なら検出して例外、許可するなら採用規則を明示してください。」
**テスト**：一意ID、重複ID、重複時の優先規則。

### 問060
**判定：Request changes（二回利用が必要な場合）**
**回答例**：「ジェネレーターは一度目の走査で消費されるため、二回目は空になります。二回必要なら再生成するか、規模を確認した上でリスト化してください。」
**テスト**：二回目にも期待データが必要なケース。

## 第7章：ファイル・JSON・入力

### 問061
**判定：Comment（移植性要件があればRequest changes）**
**回答例**：「文字コードを省略すると環境の既定値に依存し、別OSでデコード失敗または文字化けします。仕様がUTF-8なら `read_text(encoding='utf-8')` としてください。」
**テスト**：日本語UTF-8と不正バイト列。

### 問062
**判定：Request changes（重大）**
**回答例**：「入力に `../../` や絶対パスを指定されると、許可ディレクトリ外のファイルを削除できます。解決後のパスが許可ルート配下か検証し、対象も許可リストで制限してください。」
**テスト**：正常相対パス、`../`、絶対パス、シンボリックリンク。

### 問063
**判定：Request changes（辞書前提で利用する場合）**
**回答例**：「JSONのトップレベルは配列や文字列にもなれます（ほかに数値、真偽、null）。解析後に `dict` であることを検証してください。」
**テスト**：オブジェクト、配列、文字列、null。

### 問064
**判定：Request changes（重要設定の場合）**
**回答例**：「直接上書き中に停止すると設定ファイルが空または途中までになり得ます。同じファイルシステムの一時ファイルへ書き、flush/fsync後に原子的に置換してください。」
**テスト**：書込み途中の例外でも旧ファイルが残ること。

### 問065
**判定：Request changes**
**回答例**：「`'Minemura, Kazuki'` のような引用符内のカンマまで区切られ、列がずれます。標準の `csv` モジュールを使用してください。」
**テスト**：引用符付きカンマ、改行、引用符のエスケープ。

### 問066
**判定：Comment（巨大入力が現実にあるならRequest changes）**
**回答例**：「巨大ファイルを一括で読むとメモリ不足や長い停止を起こします。行単位の反復、チャンク読込み、ストリーミング解析を使ってください。」
**テスト**：想定最大サイズでメモリと時間を計測する。

### 問067
**判定：Request changes（競合があり得る場合）**
**回答例**：「存在確認とopenの間に削除・置換されるTOCTOU競合があります。先に確認せず直接openし、`FileNotFoundError` を処理してください。」
**テスト**：存在する、存在しない、アクセス権なし。

### 問068
**判定：Request changes（外部入力の場合）**
**回答例**：「拡張子は変更できるため画像内容を保証しません。サイズ上限を設け、画像ライブラリでデコードして形式・寸法を検証してください。」
**テスト**：正常画像、拡張子偽装、破損画像、巨大画像。

### 問069
**判定：Request changes**
**回答例**：「解析失敗を空設定へ変換すると、破損に気づかず既定値で起動し、誤接続や設定上書きを起こします。エラーを明示して安全側に停止してください。」
**テスト**：破損時に起動または更新を中止すること。

### 問070
**判定：Comment（頻繁ならRequest changes）**
**回答例**：「失敗のたびに一時ファイルが残るとディスクを圧迫し、秘密情報も残存します。`finally` で削除するか、一時ファイル管理機能を使ってください。」
**テスト**：成功時・例外時の双方で不要ファイルが残らないこと。

## 第8章：HTTP・外部API

### 問071
**判定：Request changes**
**回答例**：「相手が応答を返さないとワーカーが長時間占有され、処理待ちが連鎖します。接続・読取りのtimeoutを要件に基づいて指定してください。」
**テスト**：接続不能、遅延応答、正常応答。

### 問072
**判定：Request changes**
**回答例**：「500は成功データではないため、JSON本文を通常結果として返すと誤処理します。先にHTTPステータスを確認し、成功時だけ本文を検証してください。」
**テスト**：200、400、500、JSONでないエラー本文。

### 問073
**判定：Request changes（外部由来IDの場合）**
**回答例**：「IDに `/`、`?`、`#` などが含まれるとURL構造が変わります。パス要素としてエンコードするか、クエリならHTTPライブラリの `params` を使用してください。」
**テスト**：空白、スラッシュ、非ASCII、クエリ記号。

### 問074
**判定：Request changes（重大）**
**回答例**：「ソース内トークンはGit履歴、配布物、ログから漏えいします。直ちに失効・再発行し、秘密管理サービスまたは環境変数から注入してください。」
**テスト**：リポジトリとログに秘密値がなく、未設定時は安全に失敗すること。

### 問075
**判定：Request changes**
**回答例**：「429で即時再試行すると制限を悪化させます。`Retry-After` とAPI固有のレート制限ヘッダーを参照し、待機と回数上限を設けてください。」
**テスト**：Retry-Afterあり・なし、上限到達。

### 問076
**判定：Request changes**
**回答例**：「timeoutでもサーバー側ではPOSTが完了している可能性があり、再送で重複作成します。冪等性キーを付けるか、結果照会後に再試行してください。」
**テスト**：処理完了後に応答だけ失われたケース。

### 問077
**判定：Request changes**
**回答例**：「レスポンスには個人情報、アクセストークン、Cookie、カード情報などが含まれ得ます。許可した項目だけを記録し、秘密値をマスクしてください。」
**テスト**：エラー時を含め、ログに機密値が出ないこと。

### 問078
**判定：Request changes**
**回答例**：「プロキシや障害ページがHTMLを返すとJSON解析に失敗し、本来の障害原因が隠れます。ステータスと `Content-Type` を確認してから解析してください。」
**テスト**：JSON成功、HTMLエラー、Content-Type不一致。

### 問079
**判定：Request changes**
**回答例**：「必須キーが欠落すると後段で `KeyError` になり、API変更なのかデータ不正なのか分かりません。境界でスキーマ・型・必須キーを検証してください。」
**テスト**：正常、キー欠落、型違い、余分なキー。

### 問080
**判定：Comment（運用要件があればRequest changes）**
**回答例**：「接続エラーは基盤・ネットワーク障害、404は対象不存在で対応者と再試行方針が異なります。同じ表示では利用者対応と監視の切り分けを誤るため、内部エラー分類を分けてください。」
**テスト**：接続不能、timeout、404。

## 第9章：テストの読み方

### 問081
**判定：Comment**
**回答例**：「最初の異常系として、必須入力が欠落した場合に定義済みの入力エラーになるテストを追加してください。」
**確認**：対象機能で最も起こりやすく影響の大きい失敗を選ぶ。空入力や境界値も候補。

### 問082
**判定：Comment**
**回答例**：「`test_1` では失敗時に期待動作が分かりません。例えば `test_divide_raises_value_error_when_divisor_is_zero` のように条件と期待結果を示してください。」
**確認**：テスト名だけで失敗理由を推測できること。

### 問083
**判定：Comment**
**回答例**：「実行時刻によって日付・期限の境界をまたぎ、同じコードでも成功と失敗が変わります。時計を注入し、固定時刻でテストしてください。」
**テスト**：境界の直前・一致・直後。

### 問084
**判定：Comment**
**回答例**：「外部APIの障害、速度、データ変更、利用制限により単体テストが不安定・低速になります。HTTPクライアントを差し替え、契約した応答を使ってテストしてください。」
**確認**：実接続は別の結合テストで少数実施する。

### 問085
**判定：Comment**
**回答例**：「`assert result` は真らしさしか見ず、誤った数値や文字列でも通ります。型と具体的な期待値を `assert result == expected` で確認してください。」
**テスト**：誤値なら確実に失敗することを確認する。

### 問086
**判定：Comment**
**回答例**：「10動作を一つで確認すると、最初の失敗で後続が実行されず原因と未確認範囲が分かりません。独立した振る舞いごとにテストを分けてください。」
**確認**：ただし一連のシナリオを確認する結合テストまで機械的に分割しない。

### 問087
**判定：Comment**
**回答例**：「seedが毎回異なると、失敗入力を再現できず調査や修正確認が困難です。seedを固定し、失敗時にseedを出力してください。」
**確認**：複数seedの探索は別テストやプロパティテストとして行う。

### 問088
**判定：Comment**
**回答例**：「例外なら何でも通るテストでは、別のバグによる `TypeError` でも成功します。期待する例外型と、契約上重要なメッセージまたは属性を確認してください。」
**テスト**：期待外の例外では失敗すること。

### 問089
**判定：Comment**
**回答例**：「内部呼出回数だけを厳密に固定すると、結果を変えない最適化や関数分割でもテストが壊れます。契約上重要な出力・副作用を中心に確認してください。」
**確認**：外部送信回数など、回数自体が要件なら検証を残す。

### 問090
**判定：Comment**
**回答例**：「1と100だけでは範囲内の通常値を処理できるか分かりません。代表値として50を追加し、入力検証なら範囲外の0と101も追加してください。」
**テスト**：0、1、50、100、101。

## 第10章：レビュー判断

### 問091
**判定：Comment（Nit / Low）**
**回答例**：「意味が十分伝わり規約違反でもないなら、好みの違いは動作や保守性へほぼ影響しません。修正必須にせず `Nit:` として任意提案にします。」
**判断**：チーム規約違反や誤解を生む名前なら重要度を上げる。

### 問092
**判定：Request changes（Critical / High）**
**回答例**：「認証情報がログ閲覧者や外部ログ基盤へ漏れ、アカウント不正利用につながります。マージを止め、出力を除去し、既に出た資格情報は失効してください。」
**テスト**：正常時・例外時ともログに秘密値がないこと。

### 問093
**判定：通常はComment**
**回答例**：「本当に到達不能なら現時点の実行障害ではなく、読解を妨げ将来古い処理が復活する保守性問題です。ただし到達するはずの要件があるなら制御フローのバグなのでRequest changesです。」
**確認**：入力条件と分岐を示して到達可能性を判断する。

### 問094
**判定：提案を却下または任意Comment**
**回答例**：「複数実装や差し替え要件がなく、5行で明確ならクラス化は理解コストを増やします。具体的な変更要求が出るまでは関数のままにします。」
**判断**：将来使うかもしれない、だけでは複雑化の根拠にしない。

### 問095
**判定：Comment（値の根拠を確認）**
**回答例**：「timeout値は外部APIの通常・p95/p99応答時間、利用者の待機上限、呼出回数、再試行回数、上位処理の期限から決めます。根拠と計測値をPRへ記載してください。」
**テスト**：期限内成功、timeout、リトライ込みの総時間。

### 問096
**判定：根拠不足のため修正要求にしない**
**回答例**：「どの入力・状態で、期待値と実際値がどう異なり、何に影響するかが不足しています。最小の再現例または失敗テストを示してください。」
**判断**：再現できない推測を重大バグとして断定しない。

### 問097
**判定：Request changesを保留し仕様確認**
**回答例**：「提案どおり直すと元の仕様を変更します。要件、利用者、互換性、移行方法を確認し、合意した期待動作をテストにしてから実装してください。」
**判断**：レビュー担当者の推測だけで仕様を変更しない。

### 問098
**判定：原則却下または計測依頼**
**回答例**：「ボトルネックの証拠がなく、複雑性と不具合リスクだけが増えるため現時点では採用しません。代表負荷で計測し、目標との差を示してから再検討してください。」
**確認**：性能目標、現在値、変更後、保守コスト。

### 問099
**判定：Request changes（修正は必要、導入方法を検討）**
**回答例**：「脆弱性の悪用可能性と影響、壊れる利用者、代替手段、段階移行、告知、緊急度を確認します。危険な互換性を無期限に残さず、安全な移行計画を作ってください。」
**テスト**：攻撃ケース、新仕様、移行期間の互換経路。

### 問100
**判定：全件を同列に扱わない**
**回答例**：「①影響度（漏えい・停止・データ損失）、②発生可能性と再現条件、③変更箇所との関連性、④修正コストと副作用で並べます。まず確実なCritical/Highだけをマージ条件にします。」
**判断**：重複・好み・根拠不足を除外し、重要な指摘が埋もれないようにする。

## 採点方法

各問4点です。

- 1点：コード上の事実を正しく説明した
- 1点：問題が起きる入力・状態を示した
- 1点：利用者・運用・保守への影響を示した
- 1点：実行可能な修正、テスト、または仕様確認を提案した

判定が模範例と異なっても、前提条件と理由を説明できていれば減点しません。実務では仕様と変更範囲によって判定が変わるためです。
