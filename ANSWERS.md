# コードレビュードリル回答例

この回答は一例です。システムの用途、データの機密性、再実行要件などによって、判定や修正方針は変わります。

## ドリル1：外部API通信

**判定：却下**

**重大度：Critical**

### 問題点

1. 認証トークンがソースコードへ直接記述され、漏えい・更新不能の危険がある。
2. タイムアウトがなく、外部サービスが応答しないと処理が無期限に停止し得る。
3. `raise_for_status()` 相当の確認がなく、401や500の応答も正常系としてJSON解析する。
4. `opportunity_id` をURLへ直接連結しており、パス操作文字や不正なIDを拒否しない。
5. Content-Type、JSON解析エラー、トップレベルの型を検証していない。
6. URL、認証情報、HTTPクライアントが関数内に固定され、単体テストしづらい。

### 発生条件

- 外部CRMサービスが応答しない場合、タイムアウトを指定していないため待機が長期化する。
- 401の場合、エラーを明示せず、エラー本文を辞書として返すかJSON解析で失敗する。
- HTMLの場合、`response.json()` がJSON解析例外を送出する。
- `../users` はそのままURLへ入り、HTTPクライアントやサーバーの正規化次第で意図しないパスへ到達し得る。
- JSON配列の場合、型注釈の `dict` と異なる `list` を返す。

### 想定される影響

認証情報漏えい、ワーカースレッド枯渇、誤ったエラー処理、意図しないリソース参照、呼び出し側の型エラーが発生する。

### 修正方針

トークンは秘密情報ストアまたは実行環境から注入する。IDを許可形式で検証し、URLパスは安全に組み立てる。接続・読み取りタイムアウト、HTTPステータス確認、Content-Type確認、JSON例外処理、トップレベルが辞書であることの検証を追加する。ベースURLとHTTPクライアントは依存性注入し、例外方針を呼び出し側との契約として定める。

### 追加すべきテスト

- 正常な辞書レスポンス
- 接続タイムアウトと読み取りタイムアウト
- 401、404、429、500
- HTMLおよび壊れたJSON
- `../users`、空文字、許可長超過のID
- JSON配列、`null`、必須キー欠落
- ログや例外へトークンが出ないこと

## ドリル2：例外を握りつぶすコード

**判定：却下**

**重大度：High**

### 問題点

1. `Exception` ですべてを捕捉し、権限不足、I/O障害、文字コード、JSON破損、プログラム不具合を区別できない。
2. 壊れた設定でも空設定で継続し、安全でないデフォルト動作を招く。
3. 文字コードが明示されず、実行環境により結果が変わる。
4. JSONのトップレベルが辞書か検証していない。
5. エラー情報が失われ、運用時に原因を特定できない。

### 発生条件

ファイル不在だけでなく、不正JSON、アクセス拒否、ディレクトリ指定、読み取り障害、デコード失敗でも `{}` を返す。

### 想定される影響

必須設定なしで処理が続行され、誤送信、誤接続、セキュリティ設定の無効化など、本来停止すべき障害が別の障害として遅れて現れる。

### 修正方針

`FileNotFoundError` の場合だけ明示されたデフォルト設定を返す。`encoding="utf-8"` で読み、不正JSONはファイルパスと解析位置を含む例外として停止する。読み取った値が辞書でなければ拒否する。メッセージにパスと原因は含めるが、設定値や秘密情報は含めない。

### 追加すべきテスト

- ファイル不在ではデフォルトを返す
- 正常なUTF-8 JSONを読む
- 壊れたJSON、配列、文字列、`null` を拒否する
- アクセス拒否とデコード失敗を握りつぶさない
- エラーにパスと原因があり、秘密値がない

## ドリル3：ログへの秘密情報漏えい

**判定：却下**

**重大度：Critical**

### 問題点

1. Authorizationヘッダーからアクセストークンが漏れる。
2. URL全文からセッションIDと顧客情報が漏れる。
3. レスポンス全文から氏名、メールアドレス、アクセストークンが漏れる。
4. Cookie、リダイレクト先、例外本文にも秘密情報が含まれ得る。
5. タイムアウト、HTTPエラー処理、安全な追跡情報がない。

### 発生条件

デバッグログが有効、ログ収集基盤へ転送、障害調査でログを共有、またはレスポンスに個人情報が含まれる場合に漏えいする。

### 想定される影響

アカウント乗っ取り、個人情報漏えい、顧客機密の露出、ログ閲覧権限を経由した権限拡大、監査・法令上の問題につながる。

### 修正方針

構造化ログに `host`、`method`、`status_code`、`duration_ms`、検証済みの `request_id` だけを記録する。URLのパスやクエリ、ヘッダー、Cookie、本文は原則記録しない。例外も安全な分類名に変換する。どうしても値を残す場合はキーの許可リストを使い、多層のマスキングを補助策とする。

### 追加すべきテスト

- token、Cookie、session_id、メールアドレス、レスポンス本文がログにない
- 必須5項目が成功・HTTPエラー・タイムアウト時にも残る
- 改行を含む追跡IDでログインジェクションが起きない
- URL解析失敗時にも安全にログを出す

## ドリル4：Playwrightの不安定な待機

**判定：却下**

**重大度：High**

### 問題点

1. 固定待機は環境が遅いと不足し、速いと無駄になる。
2. `input` と最初の `a` は対象を一意に特定しない。
3. 検索開始操作や検索完了条件が明示されていない。
4. 0件、複数件、同名商談の扱いがない。
5. ログイン切れを検出せず、ログイン画面の入力欄を操作し得る。
6. 最終遷移や詳細画面の表示完了を検証しない。

### 発生条件

ネットワーク遅延、DOM変更、広告やナビゲーションリンクの追加、同名データ、セッション切れ、検索結果の遅延表示で不安定になる。

### 想定される影響

テストの断続的失敗、誤った商談の操作、障害の見逃し、CI時間増加、実行結果への信頼低下が起こる。

### 修正方針

検索入力、結果一覧、詳細画面に安定した `data-testid` またはアクセシブルなrole/nameを設ける。検索結果ローディングの終了と結果一覧の表示を待つ。0件は明示的なNotFound、複数件は一意なIDなどで絞り込むか曖昧として停止する。ログイン画面を検出したら再認証フローへ移すか、明確に失敗させる。最後は詳細画面固有の要素とURLを確認する。

### 追加すべきテスト

- 高速・低速応答
- 0件、1件、同名複数件
- セッション切れ
- DOMに無関係なinputやlinkが追加された場合
- 遷移失敗と検索APIエラー

## ドリル5：二重実行に弱いエージェント

**判定：却下**

**重大度：Critical**

### 問題点

1. メールメッセージ単位の冪等性キーがない。
2. Case作成と返信送信の進捗を永続化していない。
3. 外部システムをまたぐため単一DBトランザクションでは原子化できない。
4. リトライ回数、最終エラー、回復不能状態が管理されない。
5. 同じメールを並行処理した際の排他・一意制約がない。

### 発生条件

ケースAではCase作成後の送信失敗から再実行するとCaseが重複する。ケースBでは送信成功後、状態保存前のクラッシュから再実行すると返信が重複する。

### 想定される影響

顧客への重複連絡、担当者の二重対応、データ不整合、監査不能、再試行を恐れて復旧できない状態になる。

### 修正方針

`email_message_id` に一意制約を設け、処理レコードへ `case_id`、`case_created_at`、`reply_sent_at`、`processing_status`、`last_error`、`retry_count` を保存する。各副作用の成功直後に状態を記録し、再実行時は完了済み工程を飛ばす。Case作成先にも外部IDによるupsertを使う。メール送信は完全なexactly-onceが困難なため、送信プロバイダーの冪等性キーまたはoutboxと重複抑止を用いる。並行取得にはロックまたは状態の条件付き更新を使う。

### 追加すべきテスト

- Case成功後に送信失敗して再実行
- 送信成功直後にクラッシュして再実行
- 同一メッセージの並行処理
- 状態保存失敗、429、タイムアウト
- 最大リトライ到達と手動復旧

## ドリル6：AIエージェントの危険なツール実行

**判定：却下**

**重大度：Critical**

### 問題点

1. LLM出力を信頼し、許可ポリシーや引数スキーマを検証せず実行する。
2. 読み取り、削除、送信、任意コマンドが同じ権限境界にある。
3. 実行ディレクトリ、対象パス、コマンド、環境変数を制限しない。
4. 破壊操作の人間承認、タイムアウト、出力上限、監査ログがない。
5. 外部文書のプロンプトインジェクションが直接ツール実行へ到達する。

### 発生条件

攻撃者が読ませた文書、Webページ、メール、またはユーザー入力にツール実行を誘導する命令を混ぜた場合、LLMが削除や送信を選択し得る。

### 想定される影響

ファイル消失、情報流出、任意コード実行、権限奪取、第三者への誤送信、証跡不足が起こる。

### 修正方針

ツールごとに型付きスキーマ、許可対象、権限、リスクレベルを定義する。任意シェルは本番では禁止し、必要な操作を限定ツールとして実装する。書き込み・送信・削除は対象と差分を提示して承認を要求し、サンドボックス、最小権限、パス境界、タイムアウト、出力制限を適用する。監査ログには実行者、承認者、ツール、正規化済み引数、結果を残すが秘密値は除く。

### 操作分類例

| 操作 | 自動実行 | 承認必要 | 禁止 |
|---|:---:|:---:|:---:|
| 許可範囲内のファイル一覧取得 | ✓ |  |  |
| 許可された非機密ログの読み取り | ✓ |  |  |
| メール下書き作成 | ✓ |  |  |
| メール送信 |  | ✓ |  |
| CRMレコード更新 |  | ✓ |  |
| ファイル削除 |  | ✓ |  |
| 任意のシェルコマンド |  |  | ✓ |

### 追加すべきテスト

- 未知ツール、欠落・余分・誤型の引数を拒否
- パストラバーサル、シンボリックリンク、許可外ディレクトリを拒否
- タイムアウト、巨大出力、子プロセスを停止
- 承認の改ざん・再利用を拒否
- 文書内の命令で権限が拡大しない
- 監査ログに秘密情報がない

## ドリル7：誤った性能測定

**判定：却下**

**重大度：High**

### 問題点

1. ウォームアップがなく、初期化やコンパイル時間が混ざる。
2. 入出力トークン数、バッチサイズ、生成条件を揃えていない。
3. 非同期GPU/NPU処理の完了前に時刻を測る可能性がある。
4. キャッシュ、モデルロード、前処理・後処理の範囲が不明確である。
5. 総件数/総時間だけで分布、p50、p95、外れ値を隠す。
6. TTFTとデコード速度を分離していない。
7. `time.time()` は経過時間計測に適した単調時計ではない。

### 発生条件

初回実行、入力長のばらつき、生成長の違い、非同期デバイス、キャッシュヒットが混在するほど値が歪む。

### 想定される影響

モデルやデバイスを誤って比較し、容量設計、SLA、コスト判断、最適化の優先順位を誤る。

### 修正方針

モデルを事前ロードし、規定回数ウォームアップする。入力長、出力上限、実出力トークン数、バッチ、温度、キャッシュ条件を固定・記録する。`perf_counter()` と必要なデバイス同期を使う。ストリーミング生成で呼び出し開始から最初のトークン受信までをTTFT、最初のトークン以降の生成トークン数を経過時間で割ってDecode TPSとする。複数回測定し平均、p50、p95、標準偏差を報告する。

### 追加すべきテスト

- 同一条件での再現性
- 入力長・出力長・バッチ別測定
- キャッシュ有無
- 初回とウォームアップ後
- 非同期同期処理の有無
- 0件入力、0トークン出力、生成失敗

記録項目には、モデル名、量子化方式、入力長、出力長、バッチサイズ、デバイス、ドライバ、ウォームアップ回数、測定回数、温度、キャッシュ利用を含める。これらが異なる結果は同条件の比較にならないためである。

## ドリル8：並行処理の競合

**判定：条件付き採用**

**重大度：High**

### 問題点

1. `processed_count += 1` は読み取り・加算・書き込みの複合操作で、実装や将来の実行環境によって更新を失い得る。
2. 例外が発生すると `list()` の構築が中断し、成功件数や未処理項目が分からない。
3. 永久停止する処理へのタイムアウトやキャンセルがない。
4. 大量入力の投入、結果保持、外部資源への同時接続がメモリや接続先を圧迫し得る。
5. `map` は入力順で結果を返すため、先頭の遅い処理が後続結果の取得を妨げる。
6. 重複itemをそのまま複数回処理する。

### 発生条件

複数スレッドが同時更新する場合、処理が例外・ハングする場合、入力が大量または重複する場合、順序要件が仕様と異なる場合に問題化する。

### 想定される影響

件数誤り、部分成功の消失、処理停止、メモリ・接続枯渇、重複副作用、再実行不能が起こる。

### 修正方針

選択は **B：各処理の結果から最後に件数を計算する**。グローバル共有状態を削除し、各結果を成功・失敗・item IDを含む値として返す。入力は必要に応じてチャンク化または投入数を制限し、タスク単位のタイムアウト、キャンセル、リトライ方針を定める。重複を許すか排除するか、結果順を入力順にするか完了順にするかを仕様化する。Lockはどうしても共有カウンターが必要な場合だけ使う。

### 追加すべきテスト

- 1万件と空入力
- 一部例外と部分成功件数
- ハングする処理のタイムアウト
- 同一itemの重複
- 完了順と入力順の要件
- 高並行度での資源上限

## 実践ドリル：AI指摘の判定例

| AIの指摘 | 自分の判定 | 理由 |
|---|---|---|
| 外部APIにタイムアウトを設定する | 採用 | 障害時の無期限停止を防ぐため。 |
| 認証トークンを平文の設定ファイルへ移す | 不採用 | ソース直書きは廃止するが、平文ファイルではなく秘密情報ストアを使う。 |
| 小規模な設定読込処理に抽象基底クラスを導入する | 不採用 | 現状の変更点が一つなら複雑性が利益を上回る。 |
| すべての外部APIエラーを10回リトライする | 不採用 | 401や入力エラーは再試行しても直らず、負荷と遅延を増やす。 |
| メール送信を非同期化する | 判断保留 | 要求スループット、失敗時の再実行、運用基盤を確認して判断する。 |
| ログからレスポンス全文を削除する | 採用 | 個人情報やトークンの漏えいを防ぐため。 |
| 共有カウンターをLockで保護する | 今回は対応不要 | 結果から集計して共有状態自体を削除できる。 |

AIの指摘は、重大度だけでなく、実際の発生条件、影響、現在の要件、修正コストを確認して採否を決めます。
