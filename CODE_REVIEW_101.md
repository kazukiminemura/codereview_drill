# Code Review 101 — 解説から始める100問

1問につき論点は原則1つです。各章は「解説付き例題→10問」の順です。

## 回答の型

事実、発生条件、影響、提案の4点を書きます。「危険です」だけでは根拠不足です。

## 第1章：名前と意図

### 解説付き例題

#### 例題1：名前から計算目的を読み取れるか

```python
def calc(a, b): return a * b
```

**着眼点**：関数名と引数名だけで、処理の目的と値の意味を説明できるか確認します。

**解説**：`a`と`b`の意味が分からず、単価×数量なのか幅×高さなのか判断できません。名前が曖昧だと、呼び出し側で引数を逆に渡してもレビューで気づきにくくなります。名前は処理の意図と単位が伝わるものにします。

**レビュー例**：単価と数量から合計金額を求める関数なら、`calculate_total_price(unit_price, quantity)` のように目的と役割を名前に含めてください。

**改善例**：`def calculate_total_price(unit_price, quantity): return unit_price * quantity`

#### 例題2：真偽値の意味を名前で表せるか

```python
active = user.last_login_at is not None
```

**着眼点**：`True`と`False`が、どの業務状態を表すのか確認します。

**解説**：`active`だけでは「有効なアカウント」「現在ログイン中」「過去に一度でもログイン済み」のどれか分かりません。この式が判定しているのは最終ログイン日時の有無だけであり、アカウントの有効性とは一致しない可能性があります。

**レビュー例**：この値が表すのはログイン経験の有無なので、`has_logged_in`のように判定内容と一致する名前にしてください。

**改善例**：`has_logged_in = user.last_login_at is not None`

## 第2章：境界値と条件分岐

### 解説付き例題

#### 例題1：境界値を含む条件になっているか

```python
if age > 20: allow()
```

**着眼点**：仕様の「以上・以下」と、コードの比較演算子が一致しているか確認します。

**解説**：20歳以上を許可する仕様なら20歳だけ拒否されます。正常な値を1つ試すだけでは見逃しやすいため、境界の直前・境界・直後を確認します。

**レビュー例**：20歳以上を許可する仕様に対して`age > 20`では20歳を拒否します。`age >= 20`へ変更し、境界値をテストしてください。

**改善例**：`if age >= 20: allow()`。テスト値は19・20・21歳です。

#### 例題2：上限だけでなく下限も検証しているか

```python
def set_retry_count(count):
    if count <= 5:
        config.retry_count = count
```

**着眼点**：入力範囲の片側だけを確認していないか探します。

**解説**：上限の5は確認していますが、`-1`も条件を通ります。後続処理が`range(retry_count)`を使う場合、負数は0回実行として扱われ、設定ミスが表面化しないまま再試行が無効になります。

**レビュー例**：再試行回数が0〜5回の仕様なら、負数も受け入れる現在の条件は不十分です。下限を含めて検証してください。

**改善例**：`if 0 <= count <= 5:`とし、-1・0・5・6をテストします。

## 第3章：None・型・戻り値

### 解説付き例題

#### 例題1：未検出時の戻り値を処理しているか

```python
user = find_user(1)
print(user['name'])
```

**着眼点**：関数が成功時以外に返す値と、呼び出し側の前提が一致しているか確認します。

**解説**：未検出時に`None`が返るなら添字アクセスで失敗します。呼び出し側だけを見て安易に空辞書へ置き換えると、「ユーザーがいない」という状態を隠すため、まず戻り値の契約と未検出時の仕様を確認します。

**レビュー例**：`find_user`が未検出時に`None`を返すなら、この添字アクセスは失敗します。未検出を通常ケースとして扱うか例外にするかを決め、明示的に分岐してください。

**改善例**：通常ケースなら`if user is None: return "not found"`のように処理してから`user['name']`へアクセスします。

#### 例題2：異なる型を同じ戻り値に混在させていないか

```python
def load_count(raw):
    if raw == "":
        return "0"
    return int(raw)
```

**着眼点**：すべての経路で戻り値の型が一貫しているか確認します。

**解説**：空文字だけ文字列`"0"`、それ以外は整数を返します。呼び出し側は値ごとに型を調べる必要があり、加算時の`TypeError`や文字列連結などの誤動作につながります。

**レビュー例**：入力によって戻り値が`str`と`int`に変わります。件数を返す契約なら、すべての経路で整数を返してください。

**改善例**：`if raw == "": return 0`とし、戻り値を`int`に統一します。

## 第4章：例外処理

### 解説付き例題

#### 例題1：例外を握りつぶしていないか

```python
try:
    save()
except Exception:
    pass
```

**着眼点**：失敗後の状態、利用者への通知、原因を追跡する情報が残るか確認します。

**解説**：保存失敗なのに成功したように継続し、原因も失われます。`Exception`はプログラム上のバグまで広く捕捉するため、想定している失敗だけを扱う必要があります。

**レビュー例**：保存失敗を握りつぶすと呼び出し側が成功と誤認します。想定例外に限定し、再送出または失敗結果の返却を行ってください。

**改善例**：`except StorageError as error:`で記録した後に`raise`し、上位層で利用者へ通知します。

#### 例題2：例外時にも後始末されるか

```python
connection = open_connection()
send_data(connection, payload)
connection.close()
```

**着眼点**：途中の処理が失敗しても、確保した資源が必ず解放されるか確認します。

**解説**：`send_data`が例外を送出すると`close`へ到達せず、接続が残ります。高負荷時に接続枯渇を起こすまで見えにくいため、正常系の動作確認だけでは不十分です。

**レビュー例**：送信失敗時に接続を閉じられません。コンテキストマネージャーまたは`finally`で解放を保証してください。

**改善例**：`with open_connection() as connection: send_data(connection, payload)`とします。

## 第5章：関数と状態

### 解説付き例題

#### 例題1：呼び出し間で状態を共有していないか

```python
def add(item, items=[]):
    items.append(item)
    return items
```

**着眼点**：関数の外から見えない状態が、呼び出し結果へ影響していないか確認します。

**解説**：可変デフォルト引数は関数定義時に一度だけ作られ、呼び出し間で共有されます。引数を省略した2回目の呼び出しには前回の値が残るため、テスト順序によって結果が変わります。

**レビュー例**：既定のリストが呼び出し間で共有されます。省略時は関数内で新しいリストを生成してください。

**改善例**：`items=None`を既定値にし、`if items is None: items = []`としてから追加します。

#### 例題2：一つの関数が複数の理由で変更されないか

```python
def register_user(data):
    validate(data)
    database.insert(data)
    mailer.send_welcome(data["email"])
```

**着眼点**：検証、永続化、通知など異なる責務が一つの関数へ固定されていないか確認します。

**解説**：この関数は入力ルール、データベース、メール仕様のどれが変わっても修正されます。すべてを機械的に小関数へ分けるのではなく、外部依存を注入し、登録処理の流れと個別の実装を分離するとテストしやすくなります。

**レビュー例**：登録処理が具体的なDBとメール送信へ直接依存しており、失敗条件を個別にテストしにくいです。永続化と通知を抽象化して注入してください。

**改善例**：`register_user(data, user_repository, notifier)`のように依存を受け取り、各役割を独立して差し替えられる設計にします。

## 第6章：コレクションとループ

### 解説付き例題

#### 例題1：走査中のコレクションを変更していないか

```python
for item in items:
    items.remove(item)
```

**着眼点**：ループ対象と、ループ内で変更する対象が同じか確認します。

**解説**：走査中に同じリストを変更すると、削除後に要素が左へ詰まり、次の要素を飛ばします。少数のデータでは結果が偶然正しく見えることがあります。

**レビュー例**：走査中に`items`自体を削除しているため、一部の要素が未処理になります。元のリストを変更せずに結果を構築してください。

**改善例**：除外条件があるなら`items = [item for item in items if should_keep(item)]`とします。

#### 例題2：集合への変換で順序を失っていないか

```python
unique_events = list(set(events))
```

**着眼点**：重複排除と同時に、入力順序など別の性質を失っていないか確認します。

**解説**：`set`は重複を除けますが、入力順序を保持する契約には使えません。イベントが発生順に表示される仕様なら、結果の並びが変わることは利用者に見える不具合です。

**レビュー例**：重複排除のために`set`へ変換すると、イベントの発生順を保証できません。最初に現れた順を維持する実装へ変更してください。

**改善例**：ハッシュ可能な値なら`unique_events = list(dict.fromkeys(events))`とし、重複を含む順序テストを追加します。

## 第7章：ファイル・JSON・入力

**着眼点**：開発環境と本番環境の違いで、読み書きの結果が変わらないか確認します。

**解説**：既定文字コードは環境依存で、同じファイルでもPCにより結果が変わり得ます。ASCIIだけのテストデータでは問題が再現しないため、日本語などを含む入力も必要です。

**レビュー例**：文字コードを省略すると実行環境に依存します。ファイル仕様がUTF-8なら明示してください。

**改善例**：`text = Path(path).read_text(encoding="utf-8")`とし、日本語と不正なバイト列をテストします。

#### 例題2：入力されたパスが許可範囲を出ないか

```python
report_path = Path("reports") / user_input

```

**レビュー例**：入力値に親ディレクトリが含まれると`reports`外のファイルを読めます。解決後のパスが許可ルート配下であることを検証してください。

**改善例**：基準パスと対象パスを`resolve()`し、`target.is_relative_to(base)`を満たす場合だけ読み込みます。

## 第8章：HTTP・外部API

### 解説付き例題

#### 例題1：外部通信の失敗条件を扱っているか

```python
response = requests.get(url)
return response.json()
```

**着眼点**：通信時間、HTTP状態、応答形式のすべてを成功前提にしていないか確認します。

**解説**：タイムアウトがないため応答待ちが長時間続く可能性があり、HTTP 500のJSONを正常データとして返す可能性もあります。JSONとして読めることと、業務上の成功は別です。

**レビュー例**：タイムアウトとHTTPエラー確認がないため、外部障害時に処理が停止またはエラー応答を正常扱いします。失敗条件を明示してください。

**改善例**：接続・読取タイムアウトを指定し、`raise_for_status()`後に`Content-Type`とJSONの必須項目を検証します。

#### 例題2：再試行で更新処理を重複させないか

```python
for attempt in range(3):
    try:
        return requests.post(url, json=order, timeout=5)
    except requests.Timeout:
        continue
```

**着眼点**：失敗したように見える通信が、サーバー側では完了している可能性を考えます。

**解説**：サーバーが注文を登録した直後に応答がタイムアウトすると、再試行で同じ注文を複数登録します。再試行回数を増やすだけでは安全にならず、更新APIには冪等性の設計が必要です。

**レビュー例**：POSTの応答タイムアウト後に同じ注文を再送すると、二重登録の可能性があります。一意な冪等キーを送信し、サーバー側で重複を拒否してください。

**改善例**：注文ごとに固定した`Idempotency-Key`を付け、同じキーの再送が同じ結果を返すことをテストします。

## 第9章：テストの読み方

### 解説付き例題

#### 例題1：期待値を具体的に検証しているか

```python
assert calculate(2, 3)
```

**着眼点**：テストが「何らかの値」ではなく、仕様上の結果を検証しているか確認します。

**解説**：真であることしか確認せず、期待値6でなく1や文字列でも通る場合があります。実装を意図的に壊したときにテストが失敗するかを考えると、検証不足を見つけやすくなります。

**レビュー例**：このアサーションはtruthyな戻り値なら通るため、計算結果の誤りを検出できません。期待値を直接比較してください。

**改善例**：`assert calculate(2, 3) == 6`とし、0や負数など仕様上重要な入力も追加します。

#### 例題2：モックが実装の誤りを隠していないか

```python
mailer.send = Mock(return_value=True)
register(user)
mailer.send.assert_called_once()
```

**着眼点**：呼び出した事実だけでなく、正しい相手へ正しい内容を渡したか確認します。

**解説**：このテストは`send`が1回呼ばれれば通るため、宛先が別ユーザーでも本文が空でも検出できません。呼び出し回数は重要ですが、外部へ渡す値も契約の一部です。

**レビュー例**：回数だけの検証では誤った宛先や内容を見逃します。期待する引数も検証してください。

**改善例**：`mailer.send.assert_called_once_with(user.email, expected_message)`とし、送信不要な条件では未呼び出しも確認します。

## 第10章：レビュー判断

### 解説付き例題

#### 例題1：複雑な提案に直す価値があるか

```text
AI提案: 5行の関数をFactoryとStrategyに分割する
```

**着眼点**：指摘が技術的に成立するかだけでなく、変更の利益がコストとリスクを上回るか判断します。

**解説**：複数実装などの要件がなければ、抽象化によってファイル数や追跡箇所が増えるだけです。SOLID原則はクラスを増やす規則ではなく、変更理由や依存方向を整理するための判断材料です。

**レビュー例**：現時点で実装の差し替え要件がなく、抽象化の利益が示されていません。将来要件が具体化するまでは現在の単純な実装を維持します。

**改善例**：不具合、変更頻度、複数実装の予定を確認し、根拠がなければAI提案を不採用にします。

#### 例題2：実害の大きさから優先度を決められるか

```python
logger.info("login token=%s", access_token)
variable_name = "usr"
```

**着眼点**：同じ変更内に複数の指摘があるとき、好みではなく影響と緊急性で優先順位を付けます。

**解説**：`usr`の命名改善は保守性の課題ですが、トークンのログ出力は認証情報の漏えいにつながります。指摘数や修正の簡単さではなく、悪用可能性と影響範囲から重大度を決めます。

**レビュー例**：アクセストークンがログへ出るため、この変更は`Request changes`です。トークン出力を削除し、既存ログへの漏えいも確認してください。命名は別の`Comment`として扱えます。

**改善例**：秘密情報をログ項目から除外し、ログマスキングのテストと、漏えい時のトークン失効手順を用意します。
